# 0.7.7 runner/worker cutover handoff

This note is a handoff summary for the `0.7.7` runner/worker cutover work described in `notes/0.7.7-runner-worker-cutover-plan.md`.

## Status

The cutover has been implemented in the current working tree and validates with:

```bash
swift test
```

Final validation result in this session:
- `228 tests in 31 suites passed`

## What was implemented

### 1. Runner protocol cutover to v11

The mux runner protocol was converted from blocking `runBash` RPCs back to callback-based persistent bash.

Implemented surface:
- `startBash`
- `cancelBash`
- `bashHeartbeat`
- `bashFinished`

Key files:
- `Sources/WuhuCore/Runner.swift`
- `Sources/WuhuCore/RunnerCallbacks.swift`
- `Sources/WuhuCore/RunnerProtocol.swift`
- `Sources/WuhuCore/MuxRunnerCodec.swift`
- `Sources/WuhuCore/MuxRunnerHello.swift`
- `Sources/WuhuCore/MuxRunnerClient.swift`
- `Sources/WuhuCore/MuxRunnerHandler.swift`
- `Sources/WuhuCore/RunnerServerHandler.swift`

Notes:
- protocol version is now `11`
- `bashOutput` was **not** restored; heartbeat is the live-only liveness signal
- final results still travel through `bashFinished`

### 2. Worker subsystem restored

The pre-`0.7.5` worker model was restored in adapted form:
- worker generations named by epoch
- previous generations discovered and adopted
- dead generations scavenged from disk
- current generation spawned and respawned on disconnect
- old generations drain and are removed

Implemented files:
- `Sources/WuhuCore/WorkerDiscovery.swift`
- `Sources/WuhuCore/WorkerSpawner.swift`
- `Sources/WuhuCore/WorkerManager.swift`
- `Sources/WuhuCore/WorkerCallbackBuffer.swift`
- `Sources/WuhuCore/WorkerCallbackForwarder.swift`
- `Sources/WuhuRunner/MuxWorkerConnector.swift`
- `Sources/wuhu/WorkerCommand.swift`

### 3. Local runner topology restored

The local runner is out-of-process again:

server -> local runner child over UDS mux -> worker manager -> worker(s)

Implemented files:
- `Sources/WuhuServer/WuhuLocalRunnerSpawner.swift`
- `Sources/WuhuRunner/WuhuMuxRunnerServer.swift`
- `Sources/WuhuServer/WuhuServer.swift`
- `Sources/WuhuServer/WuhuServerConfig.swift`
- `Sources/wuhu/WuhuCLI.swift`

Notes:
- `local_runner_socket` support was added to server config
- `worker` CLI subcommand was restored
- `RunnerRegistry` was adjusted so the spawned local runner can replace/remove the local entry cleanly instead of relying on a permanently built-in local runner

### 4. Persistent bash reconnected to HEAD runtime

The `bash` tool now goes through `Runner.startBash` in normal session execution and returns a pending state to the loop rather than blocking until completion.

Implemented pieces:
- `PendingToolExecutionError` used as the internal pending sentinel
- `AgentLoop.executeToolCalls` updated so pending external completion does **not** immediately persist a tool failure and does **not** continue to inference
- delivered bash results are later persisted via runtime callback routing

Key files:
- `Sources/WuhuCore/CodingAgentTools.swift`
- `Sources/WuhuCore/PendingToolExecutionError.swift`
- `Sources/WuhuCore/BashToolResultFormatting.swift`
- `Sources/WuhuCore/Contracts/AgentLoop.swift`
- `Sources/WuhuCore/Contracts/AgentLoopContracts.swift`
- `Sources/WuhuCore/WuhuSessionBehavior.swift`
- `Sources/WuhuCore/WuhuSessionRuntime.swift`
- `Sources/WuhuCore/WuhuService.swift`

Important behavior:
- normal session `bash` calls are persistent / callback-driven
- direct local fallback still exists in `CodingAgentTools` for non-session/local situations where no mount resolver is present
- `async_bash` was left unchanged

### 5. Heartbeat path and server-owned lost detection

Heartbeats now flow:
- worker -> runner -> server

Server-side source of truth:
- heartbeats refresh `tool_call_status.updatedAt`
- stale recovery checks `updatedAt`
- stale/lost deadline is ~60s

Key files:
- `Sources/WuhuCore/WuhuSessionBehavior.swift`
- `Sources/WuhuCore/WuhuSessionRuntime.swift`
- `Sources/WuhuCore/WuhuService.swift`
- `Sources/WuhuCore/SQLiteSessionStore.swift`
- `Sources/WuhuCore/ToolCallRecord.swift`

Implementation details:
- `ToolCallRecord(status, updatedAt)` was reintroduced in the runtime/store state
- `AgentLoop.send(..., wakeLoop: false)` was added so heartbeats can refresh state without spuriously kicking the loop
- stale detection only considers `.started` calls whose `updatedAt` is older than the deadline and that still have no persisted tool result

### 6. Startup recovery restored

Previously-running sessions are resumed after runner callback routing is established.

Implemented in:
- `Sources/WuhuCore/WuhuService.swift`
- `Sources/WuhuServer/WuhuServer.swift`

Behavior:
- server startup now starts runner infrastructure first
- then starts service background routing
- then resumes previously-running sessions
- resumed sessions can receive callback-delivered bash completions and persist them into transcript

### 7. Worker callback buffer behavior

The old quadratic output append path was **not** restored.

Current behavior:
- worker buffer only persists durable final `bashFinished` results
- heartbeats are live-only and dropped while disconnected
- no disk-based output chunk append loop exists, so the old `WorkerCallbackBuffer` O(nÂ˛) append problem is avoided entirely

This is consistent with the agreed scope because `bashOutput` streaming was not required for the cutover.

## Behavioral notes / important design points

### No backward compatibility

This is a clean cutover:
- only protocol v11 is supported
- old blocking `runBash` RPC behavior is removed from the runner path

### No process-list RPC

No process enumeration / tracked-process listing protocol was added.
Recovery relies on:
- adoption of surviving workers
- heartbeats for running tags
- server-side stale detection using persisted timestamps

### Explicit timeout behavior preserved

Per-call timeout still terminates the process when requested.
There is no generic 1h bash hard limit.

### Server remains source of truth for loss

The server decides a bash is lost if:
- status is `.started`
- no final tool result exists
- heartbeat has not refreshed `updatedAt` for ~60s

### Stop behavior

`stopSession` now tries to cancel in-flight persistent `bash` tasks before repairing/persisting stopped state.
