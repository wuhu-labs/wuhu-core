# 0.7.7 runner/worker cutover plan

## Goal

Ship a clean `0.7.7` cutover that restores persistent `bash` through the runner/worker architecture from pre-`0.7.5`, but on top of current `main` / current session runtime architecture.

There are **zero deployed instances** using the old runner wire protocol, so this should be a **clean protocol cutover**, not a compatibility exercise.

## Agreed decisions

- **Protocol version:** bump to **11**.
- **`bash` tool:** restore **persistent / worker-backed** behavior.
- **`async_bash`:** leave as-is for this cutover.
- **Local runner:** restore the **server -> local runner child -> worker(s)** topology.
- **Keep stderr logging** where useful for debugging the cutover.
- **No process-list RPC** from worker to server.
- **No generic 1h bash hard limit.**
- **Explicit tool timeout** still kills the process when requested.
- **Server is source of truth** for when a bash task is considered lost.
- **Heartbeat-driven lost policy:** if a started bash has **no heartbeat for ~60s**, the server may mark it lost / repair it.

## Important behavior model

We do **not** want a “list all tracked processes” protocol.

Instead:

1. On server start / reconnect, adopt old workers as soon as possible.
2. Worker sends periodic **heartbeat per running bash tag**.
3. Runner simply proxies heartbeat callbacks.
4. Server updates last-seen state for the tool call.
5. If heartbeats stop for ~60s, server treats the bash as lost.

Heartbeat is **not** stdout/stderr streaming. A silent command should still heartbeat.

## Scope

### In scope

- runner protocol v11
- worker process + generational adoption/drain
- persistent `bash`
- heartbeat callback path
- crash/restart recovery for persistent bash
- fix obvious regressions needed for this cutover
- fix the old `WorkerCallbackBuffer` O(n²) output append if touched anyway

### Out of scope

- backward protocol compatibility
- process enumeration RPC
- fancy worker orchestration beyond current generation/adopt/drain model
- redesign of `async_bash`
- OTEL improvements beyond preserving existing behavior

## Known issues / design notes to address during implementation

### 1. Likely stale-recovery bug from pre-0.7.5

The old system had a likely bug where persistent bash could remain suppressed from stale recovery forever because the loop kept an internal “executing” guard for a tool call that had already transitioned to pending fire-and-forget execution.

**Do not port that behavior unchanged.**

Preferred fix:

- once `startBash` returns successfully, clear any short-lived “launch in progress” guard
- rely on **heartbeats refreshing server-side last-seen time**
- stale detection should be based on **last heartbeat / last started timestamp**, not on a forever-live in-memory guard

### 2. Old `WorkerCallbackBuffer` output persistence is O(n²)

Pre-`0.7.5`, `appendOutputToDisk` re-read the whole file and rewrote it on every chunk.

If this code is restored, change it to append efficiently:

- use `FileHandle.seekToEndOfFile()` / append writes, or
- another true append path

Do not keep the quadratic behavior if it is easy to fix while porting.

### 3. Heartbeat should be live-only

Heartbeats do **not** need disk persistence.

- final `bashFinished` result must remain durable / buffered
- heartbeat can be dropped while disconnected
- once runner reconnects, the worker resumes sending fresh heartbeats for still-running tags

### 4. Use DB-backed last-seen timestamps if possible

The existing tool call status table already has `updatedAt`.

Best direction:

- heartbeats refresh the started tool call’s `updatedAt`
- stale/lost detection compares `now - updatedAt`

This makes restart behavior sane without adding a separate process-list protocol.

## Implementation strategy

Do **not** cherry-pick the old stack wholesale.

Instead:

- keep current HEAD session/runtime architecture
- surgically restore the old runner/worker model
- re-hook `bash` into HEAD’s loop with a pending-result path

## File-by-file plan

### A. Restore runner protocol v11

Bring back the callback-based bash protocol, adapted to HEAD file layout.

Likely files:

- `Sources/WuhuCore/Runner.swift`
- `Sources/WuhuCore/RunnerProtocol.swift`
- `Sources/WuhuCore/RunnerServerHandler.swift`
- `Sources/WuhuCore/MuxRunnerCodec.swift`
- `Sources/WuhuCore/MuxRunnerHandler.swift`
- `Sources/WuhuCore/MuxRunnerClient.swift`
- add back a callback types file (or equivalent) for:
  - `BashStarted`
  - `BashCancelResult`
  - `RunnerCallbacks`
  - heartbeat callback payload
  - `BashFinished`

Protocol v11 should support:

- `startBash`
- `cancelBash`
- `bashHeartbeat`
- `bashFinished`

`bashOutput` streaming is not required for this cutover unless needed as an implementation convenience.

### B. Restore worker subsystem

Port/adapt these from pre-`0.7.5`:

- `WorkerManager`
- `WorkerDiscovery`
- `WorkerCallbackBuffer`
- `WorkerSpawner`
- `MuxWorkerConnector`
- CLI `worker` subcommand

Expected behavior:

- flock / single master guard still okay to keep
- worker dirs named by generation
- old generations adopted on startup
- dead generations scavenged from disk
- current generation respawned on crash
- old generations drain and then get removed

### C. Restore local runner spawner topology

Port/adapt:

- `WuhuLocalRunnerSpawner.swift`
- `WuhuRunner/WuhuMuxRunnerServer.swift`
- `WuhuServer/WuhuServer.swift`
- optional config support in `WuhuServerConfig.swift` (`local_runner_socket`)
- `wuhu/WuhuCLI.swift` to restore `worker` subcommand

Target topology:

- server spawns local runner over UDS
- local runner owns WorkerManager
- WorkerManager spawns/adopts workers
- local runner reconnects to server if needed

### D. Reconnect persistent bash to HEAD runtime

This is the main adaptation work.

Current HEAD made `bash` blocking again via `runBash`. That must be reversed.

Need to reintroduce a pending-result path in the new loop/runtime.

Likely files:

- `Sources/WuhuCore/AgentToolTypes.swift`
- `Sources/WuhuCore/CodingAgentTools.swift`
- `Sources/WuhuCore/WuhuService.swift`
- `Sources/WuhuCore/WuhuSessionRuntime.swift`
- `Sources/WuhuCore/WuhuSessionBehavior.swift`
- possibly `SQLiteSessionStore.swift`

Target behavior:

- `bash` tool calls `runner.startBash(tag: toolCallID, ...)`
- tool returns **pending**, not final result
- later callback routes by `toolCallID`
- result is persisted into transcript after callback arrives

### E. Add heartbeat path

Heartbeat should travel:

- worker -> runner -> server

The server should refresh last-seen state for the started tool call.

Preferred implementation:

- heartbeat callback updates the tool call’s `updatedAt` without changing status from `.started`
- stale recovery marks a started bash as lost if `updatedAt` is older than ~60s

Avoid introducing a separate process registry RPC.

### F. Restore startup recovery

Bring back the old startup sequencing, adapted to HEAD:

1. start runner infrastructure
2. wire callback routing
3. resume previously-running sessions
4. avoid prematurely repairing started persistent bash calls

HEAD currently lost the old `resumeRunningSessions()` path; restore equivalent behavior.

## Suggested concrete design for heartbeat on HEAD

If possible, represent tool call state in memory with timestamps again, not just enum status.

If that is too invasive, ensure at minimum that:

- the DB `updatedAt` is refreshed on heartbeat
- stale detection reads a timestamped record from persistence, not only enum state

The old pre-`0.7.5` `ToolCallRecord(status, updatedAt)` shape is a good reference.

## Validation

At minimum run:

```bash
swift test
```

And specifically ensure coverage for:

- mux runner RPC tests
- worker discovery tests
- worker manager tests
- worker callback buffer tests
- protocol/bash callback routing tests

Recommended manual checks:

1. start server
2. launch a long-running `bash`
3. restart / re-exec server process
4. confirm:
   - worker survives
   - server reconnects
   - heartbeats resume
   - final result still lands in transcript
5. confirm explicit timeout still kills the process
6. confirm stop / full service shutdown still takes everything down cleanly

## Acceptance bar for 0.7.7

- runner protocol is v11 only
- local runner is out-of-process again
- persistent `bash` works across server restart/re-exec
- old worker generations are adopted/drained
- no process-list RPC exists
- heartbeat-driven lost detection exists (~60s)
- stderr logging remains useful enough to debug issues after the cutover
